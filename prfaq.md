# PRFAQ (Press Release / FAQ) ガイド
{: .no_toc }

Amazon の **Working Backwards** 手法の中核成果物である PRFAQ について、汎用的にまとめた解説。
新機能/新製品/新サービスの企画初期に、設計や実装よりも前に「顧客視点で何が嬉しいのか」を文章で固めるためのフォーマット。

---

## 1. PRFAQ とは

**「まだ存在しない製品/機能のローンチ日のプレスリリース + 想定 Q&A」を、設計・実装よりも前に書く** という手法。
仕様や技術要件から考えるのではなく、顧客が受け取る体験から逆算 (Working Backwards) するために使う。

> もし魅力的なプレスリリースが書けないのであれば、その製品は作るべきではない。
> — Working Backwards (Colin Bryar & Bill Carr)

### 何の役に立つか

- **顧客価値で篩にかける**: 「魅力的なプレスリリースが書けない企画」を早期にボツにできる
- **議論を構造化する**: スライドと違い、ナラティブ (文章) は論理の穴と曖昧表現が露呈する
- **チームの北極星になる**: 設計/実装中に「PR に書いた体験から逸脱していないか」を一貫した基準で問える
- **隠れた前提を可視化**: Internal FAQ が「自分たちが知らないこと」をあぶり出す
- **スコープの判断材料になる**: 「これは PR に書く価値があるか?」がリリーススコープを切る軸になる

---

## 2. 構成

PRFAQ は大きく **Press Release パート** と **FAQ パート** の 2 部構成。

### 2.1 Press Release パート (1 ページ目安)

| 要素 | 内容 | 分量 |
|---|---|---|
| Heading (見出し) | 製品名を 1 行。顧客がそのまま理解できる言葉で | 1 行 |
| Sub-heading (サブ見出し) | ターゲット顧客 + 得られるベネフィットを 1 行 | 1 行 |
| Summary (要約段落) | 都市・日付付きで「何が・誰に・なぜ嬉しいか」を 1 段落 | 3〜4 文 |
| Problem (課題段落) | 顧客視点での課題。「今どう困っているか」 | 1 段落 |
| Solution (解決策段落) | この製品が課題をどう解くか。差別化点も含めて | 1〜2 段落 |
| Quotes (引用) | 経営陣の引用 + 架空の顧客の引用 (顧客の生の声) | 各 1 つ |
| Getting Started / CTA | 利用開始の手順、URL など | 1 段落 |

**書き方の鉄則**:

- **顧客が理解できる言葉だけを使う**。社内用語、技術用語、略語は避ける
- **形容詞ではなく具体を書く**。「高速」ではなく「3 秒以内」、「簡単」ではなく「3 クリックで完了」
- **顧客の引用は架空でも具体的に**。「○○さん (営業担当、3 拠点を毎日移動)」のように肩書きと文脈をつける
- **誇張せず、本当に書ける範囲だけを書く**。書けない誇張表現があるなら、それは作るべきでない機能

### 2.2 FAQ パート (PR の数倍の分量になることが多い)

External FAQ と Internal FAQ の 2 種類があり、**Internal FAQ のほうが重要**。

#### External FAQ (報道/顧客向け)

外部に開示しうる質問。プレスリリースを読んだ顧客や記者が抱く疑問を想定する。

- 価格、課金モデル
- 提供範囲 (地域、対象ユーザー、対象プラン)
- サポート方法、SLA
- 既存サービスとの違い
- プライバシー、データ取り扱い
- 利用開始手順

#### Internal FAQ (社内向け、本質はこちら)

外部には出さない質問。**自分たちが答えたくない質問ほど、ここに書く価値がある**。

**市場/競合**

- TAM (Total Addressable Market) はどれくらいか
- 既存の代替手段は何か (人力や Excel も含めて正直に)
- なぜ今やるべきか、なぜ自分たちがやるべきか
- 競合が同じものを出してきたらどうなるか

**技術/法務**

- 解くべき技術課題のトップ 3 は何か
- 新規に必要となる能力は何か
- 第三者依存 (API、データソース、ライセンス) はあるか
- 法務/コンプラ上の論点は何か (個人情報、規制業種、GDPR 等)

**経済性**

- ユニットエコノミクス (1 ユーザーあたりの収益とコスト)
- 初期投資の規模
- 損益分岐までの想定期間
- スケール時のコスト構造

**リスク**

- 成功条件は何か (これが満たされなければ続けない)
- 想定する失敗 Top 3 とその対応
- 検証すべき前提 (validate しないと進めない仮説)

**運用**

- 障害時の挙動、degraded mode はあるか
- 監視/監査ログは何を記録するか
- ロールバック戦略

## 3. 書く順序とコツ

### 3.1 推奨順序

1. **Heading / Sub-heading を仮置き** — 1 行ずつ、顧客と価値を書く
2. **Summary を 1 段落書く** — 全体像を 4 文以内で
3. **Problem → Solution の順で書く** — Solution を先に書くと顧客課題から離れやすい
4. **Internal FAQ を厚く書く** — ここで前提と弱点を全部洗い出す
5. **Quotes を入れる** — 顧客引用に違和感が出るなら、Problem/Solution に戻る
6. **External FAQ を仕上げる** — Internal の答えを基に外向けに整える
7. **Heading / Sub-heading を書き直す** — 全部書いた後の理解で 1 行を磨く

### 3.2 時間配分

- **初版は数時間で書ききる**。完璧を目指さない
- 最終版まで複数ドラフトを経て**数日〜数ヶ月**の検討期間が典型 (Amazon の AWS 例)
- 1 ドラフトごとにレビュアーを変えて読んでもらう

### 3.3 レビューの観点

レビューする側は以下を機械的にチェックする:

- [ ] Heading だけ読んで、それが何かわかるか
- [ ] Sub-heading だけ読んで、誰のためのものかわかるか
- [ ] Summary だけ読んで、なぜ嬉しいかわかるか
- [ ] 顧客の引用が、本当にその人が言いそうな言葉になっているか
- [ ] Internal FAQ に「自分たちが答えたくない質問」が含まれているか
- [ ] 数値 (価格、規模、期間) が具体的か、それとも形容詞で逃げているか
- [ ] 既存代替手段の評価が正直か (過小評価していないか)

## 4. 適用判断 (いつ使うか / 使わないか)

PRFAQ は万能ではない。**顧客に対する価値命題が論点になるとき**にだけ効く。

| 種別 | PRFAQ 適性 | 理由 |
|---|---|---|
| 新規プロダクト/サービス | 適している | 価値命題そのものが論点 |
| 大きな新機能 (顧客の使い方が変わる) | 適している | 顧客体験の質が論点 |
| 機能追加 (既存体験の延長) | 軽量版で OK | Summary + Internal FAQ だけでも有効 |
| アーキテクチャ刷新 | 通常は不要 | RFC/ADR のほうが適切 |
| パフォーマンス最適化、リファクタ | 不要 | 設計書 / 技術メモで十分 |
| バグ修正、運用作業 | 不要 | チケット記述で十分 |

**目安: 「これは誰の何を解くか」を顧客の言葉で語れる規模なら PRFAQ。純粋に内部品質改善なら別のフォーマットを使う**。

### リリーススコープを切る道具として

複数機能のうち「最初のリリースに何を含めるか」を判断するときにも有効。
**「この機能を抜いた状態でも、PR がまだ魅力的に書けるか?」** を問う。
書けないなら必須、書けるなら次フェーズに送れる、という切り分けができる。

## 5. よくある失敗パターン

1. **速さ優先で構想を浅くする**
   - 数時間で書ける軽量プロセスのはずが、結果として後段で手戻りが増える
   - → 初版こそ雑に、ただし Internal FAQ は必ず厚く書く

2. **承認狙いの営業文書化**
   - 経営層を通すために誇張する。Internal FAQ で「都合の悪い質問」を自分から書かなくなる
   - → 「答えたくない質問」を 3 つ以上ねじ込むルールにする

3. **スキル/技術起点の発想**
   - 「自分たちが作れるもの」から逆算してしまう
   - → Problem を書くときに、技術選択肢のことは一旦忘れる

4. **競合/代替手段の過小評価**
   - 「現状の代替手段は無い」と書いてしまう
   - → 人力、Excel、別ジャンルツール、自社別機能 等を必ず列挙する

5. **書いたら放置**
   - PRFAQ は実装中も「逸脱していないか」を見るための生きたドキュメントとして残す
   - → リリース前にもう一度読み返し、書いた内容と実装が一致しているか確認する

6. **架空顧客の引用が自分の願望になる**
   - 「これがあって本当に助かりました」のような無味な引用は危険信号
   - → 具体的な状況、Before/After、何が変わったかが伝わる引用にする

7. **External と Internal を混ぜる**
   - 外向けに書くべきでないリスクを External に書いたり、深刻な前提を External で軽く流したりする
   - → 線引きを明確に。Internal は社内に向けた率直な記録

## 6. テンプレート

```markdown
# [製品名] PRFAQ

## Press Release

### Heading
[製品名を 1 行で]

### Sub-heading
[ターゲット顧客] が [どんな価値] を得られる [製品カテゴリ]

### Summary
[都市], [日付] — [企業名] は本日、[製品名] を発表しました。
[製品の概要を 1〜2 文で]。
[顧客がこれにより得られる主要なベネフィットを 1〜2 文で]。

### Problem
[ターゲット顧客] は今、[具体的な課題] に直面しています。
[既存の代替手段] では [なぜそれが不十分か]。
その結果 [何が起きているか / 何を諦めているか]。

### Solution
[製品名] は [どのようなアプローチで] この課題を解決します。
[具体的な仕組み/機能を 1〜2 段落で]。
これにより [Before → After の変化]。

### Quote (経営陣)
> [価値命題を経営目線で語る引用]
> — [氏名], [役職], [企業名]

### Quote (顧客)
> [Before/After を具体的に語る引用。状況描写を含めて]
> — [氏名], [肩書き/状況]

### Getting Started
[いつから] [どこで] [どのように] 利用できる。詳しくは [URL]。

---
```
